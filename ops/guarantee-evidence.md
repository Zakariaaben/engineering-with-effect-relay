# Relay guarantee and evidence register

Use this register during design review and incident response. A Relay claim is
defensible only when its scope, assumptions, cost, executable oracle, and end
boundary are stated together. The durable database is authoritative for local
delivery state; telemetry helps locate that evidence but does not replace it.

| Claim | Scope and assumptions | Cost | Executable oracle | Boundary |
| --- | --- | --- | --- | --- |
| Unknown event input is decoded before a destination request is built. | One Relay HTTP/process boundary using the pinned Schema and handler graph. | Decoding work and an explicit invalid-input failure path. | [`c09-02-failure-injection-matrix.test.ts`](../test/c09-02-failure-injection-matrix.test.ts) injects malformed input before any destination call. | TypeScript types do not validate bytes by themselves; already-decoded values can still be constructed incorrectly inside a trusted process. |
| One retained ingestion key and request fingerprint yields one accepted event, delivery ID, and route snapshot; a different fingerprint conflicts. | All contenders use one available PostgreSQL authority and the same retained-key policy. | A transaction, unique-key contention, fingerprint storage, and retention. | [`c08-03-atomic-intake.test.ts`](../test/c08-03-atomic-intake.test.ts) proves commit, replay, conflict, rollback, and one HTTP dispatch. | This is intake idempotency, not remote exactly-once delivery. Deleting the key ends the guarantee. |
| Committed eligible work can be discovered after its original process path disappears. | The database remains durable and reachable; at least one compatible worker and reconciliation pass eventually run. | Polling/claim queries, indexes, claim lag, and duplicate-attempt risk. | [`c08-02-durable-reconciliation.test.ts`](../test/c08-02-durable-reconciliation.test.ts) loses the process-local path and later reclaims the committed intent. | No progress bound exists while the database, workers, destination, or scheduler is unavailable. |
| Admission and active destination attempts are bounded per Relay process and per configured destination. | Every local delivery enters through the supervisor and configured bounded queues/semaphores. | Rejection or waiting under pressure, capacity tuning, and process-local metrics. | [`m5-act-gate.test.ts`](../test/m5-act-gate.test.ts) exposes the unbounded-waiting failure and proves the bounded repair; [`c09-01-retry-storm-game-day.test.ts`](../test/c09-01-retry-storm-game-day.test.ts) proves healthy traffic retains capacity during a retry storm. | These limits are not fleet-wide, tenant-fair, or a remote rate limit. More processes multiply the possible load. |
| Retry is classified and finite; retry delay does not occupy an attempt permit. | The configured maximum attempts and elapsed budget are finite, and the process clock/scheduler runs. | More requests, more latency, retained state, and policy complexity. | [`m4-act-gate.test.ts`](../test/m4-act-gate.test.ts) proves ordered exhaustion with virtual time; [`c09-01-retry-storm-game-day.test.ts`](../test/c09-01-retry-storm-game-day.test.ts) proves delayed retries do not monopolize healthy capacity. | A timed-out or disconnected request may already have committed remotely. Retry can duplicate that effect. |
| Runtime resources and ordinary background fibers have one scope owner, and shutdown closes readiness before disposing that scope. | Work uses the assembled Layer graph and cooperative cancellation. | Startup acquisition, explicit shutdown integration, and interrupted in-flight work. | [`c07-12-startup-shutdown.test.ts`](../test/c07-12-startup-shutdown.test.ts) gates acquisition and proves readiness closes before active work is interrupted. | A process crash does not run finalizers. Shutdown does not retract committed local or remote effects. |
| Only the current database claim generation may apply a delivery mutation. | All workers share one available database clock/authority and use the repository's fenced mutations. | Lease renewal traffic, expiry delay, extra columns/indexes, and rejected stale work. | [`c08-05-leases-and-fencing.test.ts`](../test/c08-05-leases-and-fencing.test.ts) advances lease time and rejects the stale owner; [`m7-m8-act-gate.test.ts`](../test/m7-m8-act-gate.test.ts) exercises replacement after remote/local uncertainty. | The remote receiver does not observe Relay's fence and can accept both requests. This is local state protection, not distributed mutual exclusion. |
| A terminal attempt record and the corresponding local delivery state change commit as one database mutation. | PostgreSQL accepts the transaction and the current claim fence still matches. | Transaction contention and durable attempt-history storage. | [`m7-m8-act-gate.test.ts`](../test/m7-m8-act-gate.test.ts) fails between remote response and local mutation, then proves the accepted mutation contains both attempt and state. | The network request remains outside the database transaction; its effect cannot be rolled back. |
| Every repeat of one delivery carries the same `DeliveryId` in the destination idempotency header. | Relay preserves the ID through claims, retries, and recovery; the receiver interprets and retains the key atomically. | Receiver-side idempotency storage, retention, and contention. | [`m7-m8-act-gate.test.ts`](../test/m7-m8-act-gate.test.ts) runs the same uncertainty twice: a receiver ignoring the key applies two effects, while a receiver enforcing it applies one. | Relay provides at-least-once attempts. End-to-end exactly-once effects require a receiver contract Relay cannot create alone. |
| Retry, repair, and termination are conditional dead-letter transitions that preserve prior attempt history. | One available database authority; the delivery is still in the expected dead-letter state; repair keeps the same destination identity. | A separate operations credential, operator judgment, state conflicts, and retained history. | [`c09-03-operational-recovery.test.ts`](../test/c09-03-operational-recovery.test.ts) proves authorization separation, one-winner actions, route choice, termination, and replay through reconciliation. | The bearer fixture has no actor identity, approval workflow, fine-grained authorization, or durable administrative audit. |
| Logs and bounded metrics can navigate to a delivery's durable status and attempts. | Identifiers and bounded attributes are propagated; telemetry is exported and retained by the deployment when claimed. | Cardinality discipline, storage/export cost, sampling, and dashboard maintenance. | [`c08-12-distributed-operations.test.ts`](../test/c08-12-distributed-operations.test.ts) correlates durable status with process evidence; [`c09-03-operational-recovery.test.ts`](../test/c09-03-operational-recovery.test.ts) proves recovery leaves ordered durable history. | Logs may expire, metrics may reset or aggregate, and traces may be sampled. None is the authoritative state record. |
| The accepted components assemble into one path from authenticated intake through delivery, dead-letter recovery, manual reconciliation, inspection, and shutdown. | The configured adapters satisfy the focused contracts above. | Integration coupling and a broader regression test that diagnoses less precisely than focused tests. | [`c09-05-full-systems-gate.test.ts`](../test/c09-05-full-systems-gate.test.ts) crosses the assembled HTTP and runtime boundaries without wall-clock waits. | One composed scenario cannot establish every row. The focused oracle named in each row remains the proof for that claim. |

## Claims Relay does not make

- end-to-end exactly-once remote effects;
- fleet-wide fairness, a global concurrency limit, or a destination rate limit;
- availability or bounded recovery while an authority is unavailable;
- multi-region consensus, linearizability beyond the named database
  transactions, or immunity to database data loss;
- durable actor-level operations audit; or
- telemetry completeness.

When a requirement needs one of these properties, add the missing authority or
choose a platform whose public contract supplies it. Do not strengthen the
wording of the claim without strengthening its oracle.
